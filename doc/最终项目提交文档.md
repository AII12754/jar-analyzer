## 一、项目概述

本项目基于开源 Java 安全审计工具 Jar Analyzer 进行增强。我们的目标不是只增加几个零散的小功能，而是围绕真实静态安全审计流程，补齐以下关键能力：

1. 更精准地发现危险方法，而不是停留在方法名或字符串层面
2. 更高效地验证漏洞链，而不是依赖人工逐个打开代码确认
3. 更完整地组织攻击面识别、风险初筛、路径验证与结果交付
4. 更适合实际安全分析场景

本项目最终形成两轮成果：

1. 第一轮：桌面端分析、检索与可视化增强
2. 第二轮：浏览器工作台、安全巡航与交付链路增强

最终交付的不只是若干新功能，而是一套从导入样本、建立索引、识别攻击面、定位风险、验证链路到生成报告的完整安全分析工作流。

## 三、项目需求分析

### 3.1 原项目存在的问题

在调研 Jar Analyzer 原始能力后，我们发现其在课程项目和真实安全分析场景中主要存在以下不足：

1. 表达式搜索主要关注方法签名，难以直接定位内部调用了危险 API 的方法
2. 搜索结果噪音较大，框架方法容易淹没业务代码
3. 调用图和污点结果可解释性不足，不利于展示与答辩
4. 浏览器端缺乏完整的导入、分析、验证、交付工作流
5. 结果共享与报告交付形式不够完善

### 3.2 项目需求

基于上述问题，我们归纳出本项目的五项核心需求：

1. 支持更深层次的方法检索能力，能够感知方法内部调用关系
2. 支持更高效的结果过滤、导出与治理能力
3. 支持更直观的调用链和污点传播可视化
4. 支持浏览器端的完整安全分析流程
5. 支持更正式的结果交付形态，如独立报告页面

### 3.3 验收目标

本项目最终验收重点不是有没有新按钮，而是是否真正实现了以下目标：

1. 能更准确地发现高价值风险点
2. 能把分析过程讲清楚、展示清楚
3. 能体现技术深度，而不是简单字符串匹配
4. 能体现较完整的工程工作量与团队协作

## 四、项目总体设计

### 4.1 总体思路

本项目采用两轮增强、逐步整合的总体思路：

1. 第一轮先增强原有桌面端的搜索、过滤、可视化能力
2. 第二轮在已有分析能力基础上，建设浏览器工作台与交付链路

### 4.2 设计原则

整个项目设计遵循以下原则：

1. 尽量复用原有分析引擎与数据库结果，避免重复造轮子
2. 以结构化数据为核心，而不是依赖表面字符串匹配
3. 以完整分析工作流为导向，而不是孤立实现单点功能
4. 同时兼顾算法能力、工程实现和结果交付完整性

### 4.3 设计层次

本项目增强主要覆盖以下层次：

1. 搜索与表达式语义层
2. 数据库查询与结构化结果复用层
3. 可视化展示层
4. 浏览器前端交互层
5. HTTP 接口与聚合层
6. 报告渲染与交付层
7. 测试与文档层

从结构上看，这不是单点修改，而是一次多层联动增强。

## 六、重点成果说明：为什么这些功能不是简单实现

### 6.1 方法调用关系感知搜索

这是第一轮最能体现技术深度的功能之一。

增强前，原工具更擅长基于方法名、描述符、注解等签名信息进行检索；增强后，我们将方法内部调用关系引入表达式搜索，使搜索结果能直接体现这个方法是否真实调用了危险 API。

其核心价值在于：

1. 不再停留在表面特征，而是进入方法调用关系层面
2. 可以直接定位调用危险 Sink 的方法，缩短人工排查路径
3. 使搜索结果更适合作为后续可视化与验证的入口

### 6.2 浏览器安全巡航

安全巡航页不是简单的信息罗列，而是把原有分析结果重新组织成可驱动审计流程的安全页面。

它统一汇总了：

1. Spring Controllers
2. Servlets
3. Filters
4. Listeners
5. Spring 路由映射
6. 危险 Sink 对应的调用者结果

这让分析过程从到处点接口看结果，升级为先看攻击面、再做风险筛查、最后做链路验证。

### 6.3 内置漏洞排查不是简单字符串匹配

这是整个项目答辩中最需要重点强调的一点。

项目中确实存在字符串检索能力，但浏览器工作台新增的内置漏洞排查并不以字符串检索为主。它的核心在于：

1. 预置一组高价值危险 Sink 模式
2. 在结构化调用关系中反查调用这些 Sink 的 caller
3. 将这些 caller 作为后续分析入口继续做调用图、反编译和 DFS 验证

因此，本项目在风险排查上强调的是结构化调用关系回溯，而不是搜到某段文本就算完成。

### 6.4 调用图、污点图与 DFS 预设形成完整验证链路

本项目的一大特色是：不仅能找出疑似危险方法，还能继续验证这条链是否具有真实风险。

验证能力包括：

1. 交互式多层调用图，用于展示跨方法调用链
2. 污点传播可视化，用于展示参数流向危险 Sink 的过程
3. DFS 预设库，用于标准化常见风险方向的分析参数

这三项能力结合在一起，构成了发现线索、验证链路、展示结果的闭环。

## 十一、项目中的问题点、解决对策与结果反思

### 问题 1：原有搜索能力容易停留在表面特征

问题表现：
原有搜索更偏签名层面，无法直接定位内部调用了危险 API 的方法。

解决对策：
引入调用关系感知搜索，把方法内部调用关系纳入表达式搜索与后续风险排查。

结果反思：
这一步显著提升了搜索质量，也为后续所有验证型功能打下了基础。

### 问题 2：结果噪音大，业务代码容易被框架代码淹没

问题表现：
真实 Java 项目中框架代码占比高，普通搜索结果可读性差。

解决对策：
增加正则能力、黑白名单过滤与结果治理手段。

结果反思：
这类功能虽然看似不如算法显眼，但对工具可用性提升非常明显。

### 问题 3：调用链与污点结果不够直观

问题表现：
文本结果对实现者友好，但对老师、评审和开发并不直观。

解决对策：
重构调用图与污点传播可视化能力，提升解释性与展示性。

结果反思：
可视化让项目不只是能算出来，而是能讲清楚。

### 问题 4：浏览器端原本缺乏完整分析工作流

问题表现：
即使已有部分接口，也没有完整浏览器工作台来承接分析过程。

解决对策：
建设浏览器导入、状态回显、安全巡航、方法联动、DFS 预设与独立报告页。

结果反思：
第二轮的价值不只在新增页面，而在于重新组织了分析流程。

### 问题 5：分析结果缺少正式交付形态

问题表现：
如果只有工具界面，没有正式报告或导出能力，项目完成度会显得不足。

解决对策：
补充 CSV 导出与独立报告页，增强结果交付能力。

结果反思：
这一步让项目更像完整系统，而不是只做实验性功能。

本项目围绕 Jar Analyzer 的原有能力做了两轮系统性增强。第一轮重点强化桌面端分析、检索和可视化；第二轮重点强化浏览器工作台、安全巡航、风险初筛、链路验证与结果交付。

3. 不只是做了分析能力增强，还补齐了测试、部署、协作说明和结果交付形态
