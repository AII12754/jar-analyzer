## 一、开场（1 分钟）

各位老师好，我们小组选择的开源项目是 **Jar Analyzer**，一个用于 Java 安全审计的桌面工具，GitHub 上有 2000+ Star。

我们的贡献围绕一个核心问题：**当安全研究员面对一个几十万行代码的 Java 应用时，怎样才能从海量方法中快速定位到真正危险的代码？**

我们做了 **三方面增强**：
1. 让搜索引擎"看穿"方法内部的调用关系
2. 让搜索结果摆脱噪音干扰
3. 让分析过程变得可视化、可交互

下面我逐个介绍并现场演示。

## 零、演示环境准备（演示前完成）

### 使用的 JAR 文件

我们使用项目自带的 **springboot-test** 测试应用进行演示，它是一个精心设计的 Spring Boot 应用，内含完整的 RCE 漏洞调用链：

```
DataController.getStatus(cmd)           ← Spring Controller，Web 入口
  → Func.getCmd(cmd)                    ← 工具类，直接传递参数（无过滤）
  → RCEService.test(cmd)               ← Service 接口
  → RCEServiceImpl.doInternal(cmd)      ← Service 实现
  → RunUtil.run(cmd)                    ← 工具类
  → Runtime.getRuntime().exec(cmd)      ← 危险 Sink！
```

JAR 文件路径：`test/springboot-test/target/springboot-test-0.0.1-SNAPSHOT.jar`（17MB）

### 启动步骤

```bash

# 2. 在 GUI 的「开始」面板中：
#    - 点击「选择文件」→ 选中 test/springboot-test/target/springboot-test-0.0.1-SNAPSHOT.jar
#    - 点击「开始分析」→ 等待分析完成（约 10-20 秒）
```

分析完成后左侧文件树会出现类列表，底部日志显示分析结果。

## 二、功能一：方法调用关系感知的表达式搜索（3 分钟）

### 讲稿

先看一个实际场景。我们加载了一个 Spring Boot 应用，想找到所有可能触发**远程命令执行**的入口——也就是方法体内直接或间接调用了 `Runtime.exec()` 的方法。

在我们改之前，Jar Analyzer 的表达式搜索只能匹配方法签名——方法名、参数类型、注解这些**表面特征**。要想知道一个方法内部调用了什么，只能逐个点进去看反编译代码，效率极低。

我们新增了两个条件：
- **`containsInvoke(className, methodName)`** —— 筛选方法体内**包含**指定调用的方法
- **`excludeInvoke(className, methodName)`** —— 排除方法体内包含指定调用的方法

技术上，我们复用了数据库中已有的 `method_call_table`（方法调用关系表），通过查询 callee 列表来判断，不需要重新分析字节码，性能有保障。

### 现场演示步骤（逐步操作）

> **前置**：已用 Jar Analyzer 加载并分析完 `springboot-test-0.0.1-SNAPSHOT.jar`

**演示 1：搜索调用了 Runtime.exec 的方法**

1. 点击顶部菜单「表达式搜索」按钮，弹出 EL 表达式编辑器
2. 在右侧模板下拉列表中选择 **「搜索调用了 Runtime.exec 的方法」**
3. 编辑器自动填入：
   ```
   #method
           .containsInvoke("java.lang.Runtime","exec")
           .isPublic(true)
   ```
4. 点击「搜索」按钮
5. **预期结果**：出现 `RunUtil.run(String)` —— 这正是调用链中直接调用 `Runtime.exec()` 的方法
6. 双击该结果 → 打开反编译代码 → 指给老师看代码中的 `Runtime.getRuntime().exec(cmd)`

**演示 2：搜索调用了反射 Method.invoke 的方法**

1. 清空编辑器，手动输入：
   ```
   #method
           .containsInvoke("java.lang.reflect.Method","invoke")
           .isPublic(true)
   ```
2. 搜索 → 展示结果（Spring 框架内部有大量反射调用）
3. **讲解**："一条表达式，瞬间定位所有含反射调用的 public 方法——安全审计中反射是重点审查对象"

**演示 3：组合使用 containsInvoke + excludeInvoke**

1. 手动输入：
   ```
   #method
           .containsInvoke("java.lang.reflect.Method","invoke")
           .excludeInvoke("java.lang.System","exit")
           .isPublic(true)
   ```
2. **讲解**："排除了调用 System.exit 的方法，因为那些通常是安全的退出逻辑"

## 三、功能二：正则表达式匹配（1.5 分钟）

### 讲稿

有时候我们不确定危险方法的精确名称，但知道一个模式。比如所有以 `exec`、`invoke`、`run`、`execute` 命名的方法都可能有风险。

我们新增了 `nameRegex` 和 `classNameRegex`，支持 Java 标准正则表达式。

### 现场演示步骤

**演示 4：正则匹配危险方法名**

1. 在 EL 编辑器中输入（或从模板选择「搜索通过正则匹配危险方法名」）：
   ```
   #method
           .nameRegex("exec.*|invoke.*|run.*|execute.*")
           .isPublic(true)
   ```
2. 搜索 → 结果包括 `RunUtil.run`、以及 Spring 框架中各种 exec/invoke 方法
3. **讲解**："一个正则覆盖多种命名模式，比一个一个搜高效得多"

**演示 5：正则匹配 Controller 类**

1. 输入：
   ```
   #method
           .classNameRegex(".*Controller.*")
           .isPublic(true)
   ```
2. 搜索 → **预期结果**：只显示 `DataController`、`PageController`、`ApiController`、`LoginController` 中的方法
3. **讲解**："聚焦 Web 入口点——Controller 是 Spring 应用中用户请求的第一站"

## 四、功能三：搜索结果白名单过滤（2 分钟）

### 讲稿

搜索结果里经常混杂大量框架方法，真正的业务代码被淹没。原来只能用黑名单排除，我们加了白名单模式——只保留你关心的包。

### 现场演示步骤

**演示 6：黑名单过滤 — 排除框架代码**

1. 切换到「搜索」面板，在方法名搜索框输入 `get`，选择 LIKE 模式，点搜索
2. **预期结果**：大量结果，包含 `org.springframework`、`org.apache` 等框架方法
3. 找到左侧 **filter list** 区域，确认下拉框为 `BLACKLIST`
4. 在过滤规则文本框中输入：
   ```
   org.springframework
   org.apache
   ```
5. 重新搜索 → 框架方法全部消失，只剩业务代码

**演示 7：白名单过滤 — 只看业务代码**

1. 将下拉框切换为 **`WHITELIST`**
2. 文本框输入：
   ```
   me.n1ar4.test
   ```
3. 搜索 → **预期结果**：只显示 `me.n1ar4.test` 包下的方法（DataController、RCEServiceImpl 等）
4. **讲解**："白名单模式下只保留你指定的包，一步到位聚焦目标代码"

## 六、功能五：交互式多层调用链可视化（3 分钟）⭐

### 讲稿

这是我们最具视觉冲击力的功能。原来的调用图只展示一层，我们重写为可交互的多层图。

### 现场演示步骤

**演示 9：交互式调用链图**

1. 在搜索结果中双击 **`RunUtil.run`** 方法 → 右侧打开反编译代码
2. 确认当前选中的是 `RunUtil.run` 方法
3. 点击右上方的 **「HTML 调用图」** 按钮 → 浏览器自动打开

**在浏览器中演示：**

4. **介绍界面**：深色背景，中央橙色节点是 `RunUtil run`（当前方法），周围是它的调用者（绿色）和被调用者（红色）
5. **双击展开**：双击绿色的 `RCEServiceImpl doInternal` 节点 → 动画效果，新节点弹出
6. **继续展开**：双击 `RCEServiceImpl test` → 再双击 `DataController getStatus` → 整条调用链浮现
7. **信息面板**：单击任意节点 → 右侧面板显示完整的类名（如 `me/n1ar4/test/util/RunUtil`）、方法名和方法描述符
8. **缩放拖拽**：鼠标滚轮缩小查看全局 → 拖拽画布对准目标区域
9. **展示图例**：左下角图例说明颜色含义

**讲解话术**：
> "我们从 `RunUtil.run`——也就是直接调用 `Runtime.exec` 的方法——开始，双击逐层追溯。
> 最终追溯到了 `DataController.getStatus`，这是一个 Spring Controller、一个 Web 接口入口。
> 这意味着从 HTTP 请求到命令执行，中间只隔了 4 层调用。整个追踪过程完全可视化。"

## 七、功能六：污点传播流可视化（3 分钟）⭐

### 讲稿

最后是污点传播可视化。Jar Analyzer 有完整的污点分析引擎，但结果是纯文本。我们把它变成了一个可视化报告。

### 现场演示步骤

**演示 10：污点分析可视化**

1. 切换到 **「漏洞链」** 面板（第 11 个 Tab）
2. 在 Sink 区域填入：
   - **Sink Class**：`java/lang/Runtime`
   - **Sink Method**：`exec`
   - **Sink Desc**：`(Ljava/lang/String;)Ljava/lang/Process;`
3. Source 区域全部留空（搜索所有可能的 Source）
4. 设置 **深度**：`8`，**限制**：`5`
5. 选中 **「从 Sink 出发」**，勾选 **「搜索所有 Source」**
6. 点击 **「开始分析」** → 等待 5-15 秒
7. 结果面板出现调用链列表

**触发污点分析可视化：**

8. 打开浏览器，输入以下 URL（提前收藏好）：
   ```
   http://127.0.0.1:10032/api/taint_graph?sink_class=java/lang/Runtime&sink_method=exec&sink_method_desc=(Ljava/lang/String;)Ljava/lang/Process;&depth=8&limit=5&from_sink=true&search_null_source=true
   ```
9. **展示可视化页面**：
   - **顶部卡片**：指出 Total Chains / Tainted (红色) / Safe (绿色) 统计
   - **展开一条红色卡片**（点击标题栏）→ 看到 SVG 流程图
   - **指出颜色**：
     - 第一个绿色节点 = Source（如 `DataController.getStatus`）
     - 最后一个红色节点 = Sink（`Runtime.exec`）
     - 中间黄色节点 = 污点经过的方法
   - **指出动画**：红色箭头上有流动的粒子，表示污点数据在流动
   - **对比一条绿色卡片**（如果有）→ 灰色节点，没有红色粒子 → 说明污点被阻断
   - **展开详情区域**：底部显示每一步的分析日志

**讲解话术**：
> "这就是实际代码中的一条真实漏洞链。红色粒子从 `DataController.getStatus` 流向 `Runtime.exec`，
> 意味着 HTTP 参数 `cmd` 未经任何过滤就到达了命令执行方法。
> 中间经过了 `Func.getCmd` → `RCEService.test` → `RCEServiceImpl.doInternal` → `RunUtil.run`，
> 每一步都没有做安全检查。一张图就是一份漏洞报告。"

## 九、总结（30 秒）

|  | 原来 | 我们的改进 |
|--|------|-----------|
| 表达式搜索 | 只能匹配签名特征 | 能感知方法内部的调用关系 |
| 结果过滤 | 只有黑名单 | 黑名单 + 白名单 + 正则 |

| 调用图 | 单层静态图 | 多层交互式可视化 |
| 污点分析 | 纯文本日志 | SVG 流程图 + 动画 |

## 附录 D：springboot-test 中的漏洞调用链说明

```
完整调用链（6 层）：

DataController.getStatus(String cmd)
        ↓ 调用 new Func().getCmd(cmd) 获取命令
        ↓ 传入 rceService.test(newCmd)
RCEServiceImpl.test(String cmd)
        ↓ 调用私有方法 doInternal(cmd)
RCEServiceImpl.doInternal(String cmd)
        ↓ 调用静态方法 RunUtil.run(cmd)
RunUtil.run(String cmd)
        ↓ 调用 Runtime.getRuntime().exec(cmd) ← 远程命令执行！

这条链中没有任何输入验证/过滤，是一个典型的命令注入漏洞。
```
