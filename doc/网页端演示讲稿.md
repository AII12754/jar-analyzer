# Jar Analyzer 浏览器工作台演示讲稿

> 适用版本：当前工作区实现的浏览器工作台增强版
> 推荐样本：`test/innovation-demo/target/innovation-demo-0.0.1-SNAPSHOT.jar`
> 备选样本：`test/springboot-test/target/springboot-test-0.0.1-SNAPSHOT.jar`

---

## 一、开场（1 分钟）

各位老师好，这次我们不是单纯给 Jar Analyzer 补几个按钮，而是把它已有的静态分析能力重新组织成了一条更完整的网页端安全分析流程。

我们这次重点展示 5 件事：

1. 浏览器里直接导入样本并建立分析索引
2. 用“安全巡航”页面先看清 Web 入口和重点风险面
3. 用内置漏洞排查快速锁定危险 Sink 的调用者
4. 从线索一键进入方法工作台、调用图和 DFS 污点分析
5. 最后直接生成可交付的独立报告页面

这 5 步对应真实审计流程里的 5 个阶段：导入、摸清攻击面、缩小排查范围、验证调用链、形成交付物。

---

## 二、演示环境准备

### 推荐样本

优先使用 `innovation-demo`，因为它同时覆盖两种现实场景：

1. Spring MVC Web 入口到 `Runtime.exec` 的命令执行链
2. `ObjectInputStream.readObject` 到危险 Sink 的反序列化链

其中最适合网页端演示的一条命令执行链是：

`DemoCommandController.exec(String cmd)`
→ `DemoCommandFacade.dispatch(cmd)`
→ `DemoCommandService.execute(cmd)`
→ `DemoCommandService.doInternal(cmd)`
→ `Runtime.getRuntime().exec(cmd)`

这条链很适合讲，因为它既有真实 Web 入口，也有明确危险 Sink，而且中间跨了 Controller、Facade、Service 多层业务封装。

### 演示前准备

1. 编译主项目：`mvn clean package -DskipTests`
2. 编译样本：进入 `test/innovation-demo` 执行 `mvn clean package -DskipTests`
3. 启动 jar-analyzer 主程序并打开浏览器工作台
4. 确认浏览器页可以访问，并且还没有加载任何 JAR

## 三、功能一：浏览器导入与索引建立

### 讲稿

老师们先看第一步。以前很多静态分析工具的门槛在于，必须先理解桌面端面板、数据库状态和缓存状态，才能进入分析。我们这次把导入、构建和状态回显直接放进浏览器工作台里，目的是让分析动作从一开始就更连续。

### 实现怎么做

这里不是简单上传一个文件就结束了，背后实际做了三层工作：

1. 浏览器侧支持直接上传 JAR/WAR，也支持填写服务端本机路径
2. 服务端复用原有构建流程去生成数据库和分析索引
3. 前端轮询构建状态，把进度、日志、输入源、数据库状态和引擎状态持续反馈出来

它的难点不在“传文件”，而在于让浏览器端能稳定接管原来只在桌面里完成的构建状态流，并且保证构建结束后后续分析页面拿到的是最新索引。

### 功能是什么

浏览器中直接完成：

1. 上传样本
2. 启动构建
3. 查看构建日志
4. 确认数据库和引擎是否就绪

### 实际意义

真实项目里，审计第一步不是“找漏洞”，而是先保证样本真的被正确加载。很多误判和空结果都不是算法问题，而是样本没导入好、索引没生成完。把这一步透明化，能明显降低使用门槛，也减少误操作。

### 演示动作

1. 打开“浏览器构建”页
2. 选择 `innovation-demo-0.0.1-SNAPSHOT.jar`
3. 点击开始构建
4. 展示进度条和日志滚动
5. 构建完成后指出状态卡片里的引擎、JAR 数量、缓存状态已经刷新

现场可以这样说：

> 这一页解决的是“样本什么时候真正进入分析状态”的问题。老师现在看到的不只是上传成功，而是数据库、日志、缓存和引擎四个状态一起闭环了。

## 四、功能二：安全巡航与入口资产可视化

### 讲稿

第二步不是立刻搜 `Runtime.exec`，而是先看清这个应用到底有哪些 Web 入口。因为真实审计里，如果攻击面都没摸清，后面的漏洞排查很容易只盯住单点，忽略实际可达性。

### 实现怎么做

这次新增了一个统一的安全巡航接口，把原来分散的数据源聚合起来：

1. Spring Controllers
2. Servlets
3. Filters
4. Listeners
5. Spring 路由映射
6. 内置危险 Sink 的调用者结果

后端不是简单拼 JSON，而是对入口资产做了去重和排序，对 Spring 路由做了集中提取；前端再把这些结果组织成：

1. 概览卡片
2. 优先排查方向
3. 入口资产卡片
4. Spring 入口映射列表

这一步的复杂性在于，它把“控制器识别”“组件识别”“路由枚举”和“风险线索”放进了同一个视图，形成了一个真正能先看全局、再下钻的页面，而不是几个松散的小查询按钮。

### 功能是什么

安全巡航页能直接回答四个问题：

1. 这个应用暴露了多少 Web 入口资产
2. 有多少条 Spring 路由
3. 哪些风险方向最值得先看
4. 这些入口方法分别落在哪些类和方法上

### 实际意义

在真实业务系统里，第一轮排查通常不是从所有方法开始，而是从“外部能不能打到这里”开始。入口资产可视化的意义就在于，它把攻击面从抽象概念变成了一张可以直接讨论的面板。

如果现场需要一句更直白的话，可以这样讲：

> 以前分析像是在黑屋子里拿手电筒找东西，现在先把房间结构图打开，再决定从哪一扇门进。

### 演示动作

1. 切到“安全巡航”页
2. 点击“刷新巡航”
3. 展示入口资产概览里的 Controller、Servlet、Filter、Listener 数量
4. 往下展示 Spring 入口映射
5. 指出 `DemoCommandController.exec` 这种典型 Web 入口

推荐现场话术：

> 这一页先不谈漏洞，只谈攻击面。老师可以看到系统有哪些 Web 入口、有哪些路由、哪些地方是真正可能被外部请求打到的。

## 五、功能三：内置漏洞排查与风险优先级排序

### 讲稿

摸清入口以后，第三步就是先筛高价值线索。我们这次没有继续走“字符串模糊搜索”的路线，而是做了内置危险 Sink 的调用者排查。

### 实现怎么做

这里的关键实现点有两个：

1. 我们没有把漏洞查找建立在普通字符串匹配上，而是基于方法调用关系做调用者回溯
2. 后端内置了一组危险 Sink 模板，包括：
   - `Runtime.exec`
   - `ProcessBuilder.start`
   - `Context.lookup`
   - `ObjectInputStream.readObject`
   - `ScriptEngine.eval`
   - `URL.openConnection`
   - `Method.invoke`
   - `FileOutputStream.<init>`

也就是说，系统不是只看代码里有没有出现某个字符串，而是直接问数据库：**谁调用了这些危险方法**。

这在工程上更难，但更有价值，因为它把搜索对象从“文本片段”提升成了“真实方法节点”。这样做出来的结果更适合直接进入调用图、DFS 和代码核查。

### 功能是什么

安全巡航页会给出：

1. 每类风险的命中数量
2. 风险级别排序
3. 命中的代表性方法列表
4. 每条线索对应命中了哪个危险 Sink
5. 一键进入方法工作台或调用图

### 实际意义

这一步的现实意义很明确：

1. 它把安全研究员脑子里的经验规则，做成了批量初筛能力
2. 它能把海量方法先压缩成少量“值得先看”的类别
3. 它可以把“我怀疑这里危险”变成“这里确实调用了危险 Sink，上游再继续看”

这对真实审计特别重要，因为企业项目里最贵的不是扫描，而是人工核查时间。

### 演示动作

1. 留在“安全巡航”页
2. 先展示“优先排查方向”里的高危类别
3. 再展示“内置漏洞排查”卡片
4. 点击命中方法旁边的“送入工作台”
5. 点击另一条线索的“调用图”

推荐现场话术：

> 这里最关键的一点是，我们不是在做关键字检索，而是在做危险调用者索引。只要某个业务方法真实调用到了这些高风险 Sink，它就会被提到前面来。

## 六、功能四：方法工作台与调用图联动

### 讲稿

第四步是验证。前面的巡航和排查是“找线索”，这里要回答的是：这条线索到底是不是业务真实风险，还是只是框架噪音。

### 实现怎么做

我们把浏览器里的多个分析动作串成了一个可跳转的工作流：

1. 安全巡航页可以把某个方法直接送入工作台
2. 工作台继续调用原有接口去查：
   - 调用者
   - 被调用方法
   - 实现类
   - 父类/接口实现
   - CFR / Fernflower 反编译结果
3. 同一个已选方法还能直接生成方法调用图

这里的复杂点在于“状态衔接”：一个页面里点中的方法，要能稳定带到另一个页面继续做图、做代码查看、做上下游关系核查，而不是每次都重新手输类名和方法名。

### 功能是什么

从任意安全线索出发，可以快速完成三件事：

1. 看这是不是 Controller 能打到的业务方法
2. 看它后面到底落到了哪些危险调用
3. 看反编译代码里有没有过滤、白名单或安全判断

### 实际意义

这一步是把“有线索”变成“有证据”。

真实审计里，很多高危调用并不等于漏洞，关键要看参数是不是外部可控、有没有过滤、有没有只在安全上下文里调用。方法工作台和调用图联动的意义，就是用最短路径完成这一步人工验证。

### 演示动作

1. 从安全巡航页点击 `DemoCommandController.exec` 或其下游方法
2. 进入“方法工作台”页
3. 先展示调用图
4. 再展示调用者 / 被调用结果
5. 最后打开反编译代码，指出 `Runtime.exec(cmd)` 附近没有明显过滤

推荐现场话术：

> 前一页告诉我“这里值得看”，这一页告诉我“它为什么值得看”。从入口到危险 Sink 的关系，在这里开始被证实，而不是停留在猜测层面。

## 七、功能五：DFS 预设库与污点图验证

### 讲稿

第五步是把核查再往前推进一层，从“方法级关系”进入“路径级验证”。这次我们把 DFS 面板从一个手工填 Sink 的页面，改成了一个带预设库的分析面板。

### 实现怎么做

这里新增的是一套浏览器侧 DFS 预设库，不只是多几个按钮：

1. 每个预设都绑定了 Sink 类、方法、描述符、深度和结果条数
2. 预设既能在 DFS 面板里快速填充，也能从安全巡航页直接跳转过来
3. 填充后可以继续调用 `/api/dfs_analyze` 和 `/api/taint_graph`，生成结构化结果和图形结果

也就是说，前面的漏洞排查和后面的 DFS 不是两套脱节的功能，而是一条连续的分析路径。

### 功能是什么

当前预设覆盖的方向包括：

1. 命令执行
2. JNDI
3. 反序列化
4. 脚本执行
5. SSRF
6. 文件写入
7. 反射调用

### 实际意义

真实团队协作里，一个很常见的问题是：每个人会分析，但每个人填参数的方式都不一样。预设库的意义就在于把常见危险方向标准化，把一次次重复输入变成统一入口。

它的实际价值不只是“更方便”，而是“更稳定”：

1. 新同学上手更快
2. 常见风险排查更一致
3. 结果更容易复现和比对

### 演示动作

1. 在“安全巡航”页某个命令执行卡片上点击“套用 DFS 预设”
2. 自动跳到“DFS 分析”页
3. 展示 Sink 参数已经自动填好
4. 点击“执行 DFS”
5. 再点击“生成污点图”
6. 展示从上游方法逐步汇聚到危险 Sink 的路径

推荐现场话术：

> 这里展示的是从“发现风险类别”到“验证具体路径”的自动衔接。前面筛出来的不是死结果，而是可以继续推演的分析入口。

## 八、功能六：LLM 报告与独立报告页

### 讲稿

最后一步是交付。安全分析如果只停留在工具界面里，价值是不完整的。因为真实工作里最后一定要有报告，要能给开发、给负责人、给老师直接看。

### 实现怎么做

这里做了两件事：

1. 浏览器页保留报告生成入口和日志反馈
2. 最新报告单独缓存，并提供 `/report/latest` 独立页面

这样做的好处是，工作台负责分析过程，独立页面负责阅读和交付，两者不再混在一个面板里。并且这次我们把一些明显模板化、说明书式的多余文案去掉了，让报告页更像一个正式交付物，而不是演示页。

### 功能是什么

1. 可选择先做整项目分析再生成报告，或直接基于缓存生成
2. 日志区可以看到生成进度和关键提示
3. 报告生成后可以直接打开独立页面
4. 独立页面适合在新窗口中展示、讲解和交付

### 实际意义

这一步的意义很实际：

1. 安全人员不需要再手工整理零散截图和文字
2. 开发团队可以直接看一份结构化结果
3. 现场答辩时，也能把“分析过程”和“最终交付”区分开展示

### 演示动作

1. 切到“LLM 报告”页
2. 填写模型配置
3. 点击生成报告
4. 展示日志区输出
5. 点击“打开最新独立报告页”
6. 在新页面中展示目录、摘要卡片和正文

推荐现场话术：

> 前面的页面是在帮我分析，这个页面是在帮我交付。对于真实安全工作来说，发现问题和把问题讲清楚，缺一不可。

## 十一、老师追问时的技术细节补充

这一节不是正式主讲部分，而是给老师追问“你到底怎么实现的”时备用的。

---

### 1. 浏览器构建到底做了什么

如果老师问“浏览器里点一下开始构建，背后到底发生了什么”，建议这样回答：

1. 浏览器不是自己分析字节码，而是调用后端接口 `/api/start_project_build`
2. 如果是本地上传，文件会先进入 `jar-analyzer-download/browser-upload`
3. 后端继续复用原有的构建与建库流程，不是重新写一套分析器
4. 浏览器端再轮询 `/api/build_status`，把进度、日志、数据库状态和引擎状态持续显示出来

可以补一句更具体的话：

> 这个页面本质上是把原来桌面端的构建流程浏览器化，而不是把分析逻辑搬到前端。前端负责触发和展示，真正的索引建立仍然在服务端完成。

---

### 2. 安全巡航页的数据到底从哪里来

如果老师问“这个页面是不是前端自己拼几个请求做出来的”，可以这样回答：

1. 前端主要调用的是一个统一接口：`/api/security_overview`
2. 这个接口在后端由 `GetSecurityOverviewHandler` 做聚合
3. 聚合时直接调用已有分析引擎能力：
   - `getAllSpringC()` 取 Spring Controllers
   - `getAllServlets()` 取 Servlets
   - `getAllFilters()` 取 Filters
   - `getAllListeners()` 取 Listeners
   - `getSpringM(className)` 取某个 Controller 的路由映射
4. 后端会先做去重，再按路径、类名、方法名排序，前端只负责把结果渲染成概览卡片、风险板和入口列表

这里可以强调一个实现点：

> 安全巡航不是前端随便调用十几个接口再硬拼页面，而是后端先把攻击面和风险线索整理成统一结构，前端拿到的是一个可直接展示和交互的数据模型。

---

### 3. 漏洞排查到底是不是字符串匹配

这是最值得提前准备的一问。

标准回答建议直接说：

> 不是。网页端主打的“内置漏洞排查”不是靠字符串模糊搜索，而是基于方法调用关系数据库去找“谁调用了危险 Sink”。

可以继续展开成三层：

#### 第一层：字符串匹配确实存在，但不是主方案

项目里原本就有字符串检索能力，对应的是 `StringMapper.xml` 里的 `selectMethodByString`。它查的是 `string_table`，SQL 条件本质上是：

`WHERE s.value LIKE CONCAT('%', #{value}, '%')`

这类方式适合做辅助发现，比如搜索硬编码 URL、命令片段、类名字符串，但它只能说明“某个字符串出现在方法里”，不能证明“这个方法真的调用到了危险 API”。

#### 第二层：网页端漏洞排查用的是调用关系表

网页端安全巡航里真正用的是 `MethodCallMapper.xml` 里的 `selectCallersLike`，它查的是 `method_call_table`。它的核心含义不是“搜文本”，而是：

1. 在调用关系表里找 callee 是某个危险 Sink 的记录
2. 再把对应 caller 方法找出来

SQL 条件本质上是：

1. `callee_class_name LIKE ...`
2. `callee_method_name LIKE ...`
3. `callee_method_desc = ...`（如果描述符被指定，就做精确匹配）

所以它找的是“真实调用关系”，不是“代码里出现了相似字符串”。

#### 第三层：为什么要用这种方式

因为老师如果继续问“为什么不直接搜 Runtime.exec 字符串”，你可以这样答：

1. 搜字符串只能说明文本出现了，不能说明字节码层面真的形成了调用
2. 调用关系表已经是构建阶段提取好的结构化结果，适合做批量回溯
3. 后面还要继续接调用图、方法工作台和 DFS，所以用方法节点做结果会更稳定

一句话版本可以这样说：

> 字符串搜索解决“像不像”，调用关系表解决“是不是”。网页端这次重点做的是后者。

---

### 4. 危险 Sink 列表是怎么组织的

如果老师问“你们怎么知道哪些是高危方法”，可以这样回答：

1. 我们在 `GetSecurityOverviewHandler` 里内置了一组 `SinkPattern`
2. 每个模式包含：
   - 展示名称
   - Sink 类名
   - Sink 方法名
   - 方法描述符
3. 当前内置的方向包括：
   - `Runtime.exec`
   - `ProcessBuilder.start`
   - `Context.lookup`
   - `ObjectInputStream.readObject`
   - `ScriptEngine.eval`
   - `URL.openConnection`
   - `Method.invoke`
   - `FileOutputStream.<init>`

再补一句：

> 这些模式本质上是经验规则的结构化封装。先用固定高价值 Sink 做第一轮初筛，再让老师看到后续还能接调用图和 DFS 做验证。

---

### 5. Spring 路由和 Web 入口是怎么识别的

如果老师问“你怎么知道这个方法是 Web 入口”，可以这样答：

1. Spring 路由不是前端猜出来的，而是构建阶段已经入库
2. 路由数据在 `spring_method_table`
3. 查询用的是 `SpringMethodMapper.xml` 里的 `selectMappingsByClassName`
4. 返回字段包含：
   - `class_name`
   - `method_name`
   - `method_desc`
   - `path`
   - `restful_type`

所以安全巡航页里看到的“某个类的某个方法对应某个 HTTP 路径”，本质上是数据库里的结构化路由记录，而不是页面上手工拼出来的说明文字。

一句话版本：

> 这里用的是已入库的 Spring 映射信息，不是通过类名里有 Controller 就简单猜它是入口。

---

### 6. 方法工作台为什么能一键联动

如果老师问“为什么你在安全巡航点一个方法，下一页就能直接查调用图和代码”，可以这样回答：

1. 前端在浏览器里维护了一个 `selectedMethod` 状态
2. 在安全巡航点“送入工作台”时，实际上是把 `className`、`methodName`、`methodDesc` 写进这份状态
3. 方法工作台读取这份状态后，就可以继续调用已有接口：
   - `/api/get_callers`
   - `/api/get_callee`
   - `/api/get_impls`
   - `/api/get_super_impls`
   - `/api/cfr_code`
   - `/api/fernflower_code`
   - `/api/method_graph`

所以联动的关键不是“跳页面”，而是“把同一个方法节点在多个分析动作之间复用”。

---

### 7. DFS 预设是不是新的分析算法

如果老师问“DFS 预设是不是你们新写了一个算法”，建议明确区分：

1. 预设本身不是新的图算法
2. 预设做的是参数标准化和分析流程串联
3. 真正执行分析的仍然是原有后端接口：
   - `/api/dfs_analyze`
   - `/api/taint_graph`
4. 预设提前帮用户填好的是：
   - `sink_class`
   - `sink_method`
   - `sink_method_desc`
   - `depth`
   - `limit`
   - `from_sink`
   - `search_null_source`

可以这样总结：

> 我们这次不是重写 DFS 引擎，而是把原来需要人工理解和填写的一组参数，整理成了可复用的安全场景模板。

---

### 8. 如果老师问“你们改的到底是前端，还是后端，还是数据库”

建议直接回答“三层都动了，但分工很清楚”：

1. 前端层：新增安全巡航页、风险卡片、预设库、状态联动和页面跳转
2. 后端层：新增 `/api/security_overview` 聚合接口，把入口资产和风险线索整理成统一模型
3. 数据层：没有新造一套数据库，而是复用了已有的 `method_call_table`、`string_table`、`spring_method_table` 等结构化结果

这个回答的好处是能让老师感受到，你们不是只做了页面美化，也不是只做了一个孤立接口，而是把现有引擎、数据库和浏览器交互重新串成了一条完整流程。

### 9. 最短答辩口径

如果现场时间紧，老师只想听最核心的一句，可以直接说：

> 这次网页端增强的实现核心，不是字符串搜索，而是复用构建阶段已经生成的结构化数据库结果，尤其是 `method_call_table` 和 `spring_method_table`。我们在后端用统一接口把这些结果聚合出来，再在前端把它们组织成“入口识别、风险初筛、调用验证、DFS 验证、报告交付”的完整流程。
